# PR #2 — Protocol happiness and transport dominance framework

**Title (suggested):** `feat(perf-lab): add protocol happiness matrix, tail-weighted superiority scoring, and transport dominance modeling`

**Relationship to other PRs:** PR #1 covered capacity modeling and pool derivation (`derive-pool-sizes.js`, `capacity-one`). **This PR (PR #2)** adds protocol intelligence and dominance math. **PR #3 (future)** should wire pool env vars, ingress stream caps, and runtime policy.

---

## Executive summary

This PR introduces a **protocol evaluation framework** that:

- Computes a per-service **protocol happiness matrix** (HTTP/1, HTTP/2, HTTP/3).
- Produces **tail-weighted superiority scores** (not raw RPS alone).
- Models **transport dominance thresholds** (when HTTP/3 vs HTTP/2 matters under backend constraints).
- **Integrates** with `capacity-recommendations.json` from `derive-pool-sizes.js` so recommended pool aligns with dominance logic.

**This does not change runtime behavior.** It adds deterministic, reproducible performance **modeling** tooling only.

---

## Why this exists

We already had:

- Capacity envelopes and collapse detection.
- Recommended DB pool sizes from measured peak RPS and μ.

We did **not** yet answer, per service:

- When does HTTP/3 actually outperform HTTP/2?
- When does backend saturation mask transport gains?
- Which protocol is “best” under tail + stability, not only throughput?
- What pool size is consistent with HTTP/3 “unlocking” when τ > 0?

This PR answers those questions **mathematically** from the same ceiling `service-model.json` and performance-lab `collapse-summary.json`.

## What was added

### 1. `scripts/protocol/compute-happiness.js`

**Inputs**

- `--service-model` → ceiling run `service-model.json`
- `--collapse` → `collapse-summary.json` (default: `bench_logs/performance-lab/collapse-summary.json`)
- `--out-dir` → output directory (default: `bench_logs/performance-lab`)
- Automatically reads `capacity-recommendations.json` from `out-dir` when present (pool alignment).

**Behavior (per service)**

1. Merge duplicate protocol rows in the model (**keep max RPS** per protocol).
2. Scores (0–1 where noted):
   - `ThroughputScore = rps / max_rps` (across protocols for that service)
   - `LatencyScore = min_p95 / p95`
   - `TailRiskScore` = normalized tail penalty (p95/p50 ratio, max/p95 ratio, fail_rate term)
   - `StabilityScore`: `fail_rate > 2% → 0`; `> 0 → 0.5`; else `1`
   - `CompositeScore = 0.35·T + 0.2·L + 0.25·Tail + 0.2·Stability`
3. **Winner** = highest composite (tie-break: lower p95).
4. **Envelope** flag: winner utilization @ pool=10 &lt; 0.85 when known.
5. **Predicted safe RPS @ pool 10** from model (winner row).
6. **Collapse max RPS** from `collapse-summary.json`.
7. **Recommended pool** from capacity JSON if present, else default `10`.

### 2. Transport dominance modeling

Per service (from model + capacity):

- λ2 = HTTP/2 RPS, λ3 = HTTP/3 RPS  
- τ = λ3/λ2 − 1  
- μ = HTTP/3 **μ** (RPS per slot) for thresholding  
- `pool_threshold = λ3 / μ` (same as (λ2·(1+τ))/μ when τ is consistent)  
- B = μ × recommended_pool  
- **Dominance rule documented:** compare backend capacity B to λ2×T where T = λ3/λ2  

**Important correctness fix**

- **`h3_transport_unlocked` is false when τ ≤ 0**, with an explanatory note (HTTP/3 is not delivering higher effective throughput than HTTP/2 in the model, so pool sizing cannot “unlock” H3 transport superiority).
- When **τ > 0**, **`h3_transport_unlocked`** is **true** only if **recommended_pool ≥ ceil(pool_threshold)**.

This avoids false “HTTP/3 wins” claims when the data show λ3 ≤ λ2.

### 3. Outputs (`bench_logs/performance-lab/`)

| File | Purpose |
|------|---------|
| `protocol-happiness-matrix.json` | Per-service scores, winner, envelope, collapse RPS, recommended pool, transport_dominance block |
| `protocol-superiority-scores.json` | Detailed per-protocol score breakdown |
| `protocol-ranking.md` | Human-readable report for reviewers and lab handoff |

## Architectural impact

After this PR, for each service we can state from artifacts:

- Which protocol scores highest under the composite (throughput + latency + tail + stability).
- Whether the envelope at pool=10 looks stable (utilization gate).
- Whether HTTP/3 transport superiority is even **eligible** (τ > 0) and whether recommended pool **clears** the dominance threshold.
- When backend saturation dominates (so protocol differences are secondary).

That moves protocol discussion from opinion to **reproducible math** on the same inputs as capacity modeling.

## Reviewer FAQ (preemptive)

**Why weight tail risk at 0.25?**  
Tail amplification correlates with collapse and user-visible SLO risk; we explicitly penalize high p95/p50 and max/p95 and non-zero fail rates.

**Why not use p99 from raw k6?**  
p99 was often zero or missing in the aggregated model row; extreme tails are already characterized via lognormal(p50,p95) in `service-model.json`.

**Why HTTP/3 μ for pool_threshold?**  
The dominance band is expressed in terms of λ3 and sustained service capacity for the HTTP/3 path: threshold λ3/(μ_http3) keeps units consistent with “HTTP/3 offered load vs HTTP/3 slot capacity.”
