Middleware / logging / interceptors — what we actually have

There isn’t a single shared “Express middleware + logging helper” that every service imports. Patterns are split by layer:

services/api-gateway/src/server.ts — This is where most HTTP middleware lives: Helmet, CORS, compression, rate limiting, query sanitization, per-request Prometheus counter on finish, the JWT / identity path, http-proxy-middleware to backends, and a final Express error handler (app.use((err, ...)). Trust traffic hits /trust and /api/trust proxies after that stack.

services/trust-service/src/http-server.ts — In-process Express app only: express.json, a tiny middleware that increments httpCounter on response finish, route handlers, and requireUser (checks x-user-id). No request/response logging middleware; errors are mostly console.error + JSON body.

services/trust-service/src/grpc-server.ts — Plain gRPC handlers; no withLogging wrapper like services auth-service / messaging-service use (withLogging(handler, methodName) around RPCs). Logging here is ad hoc console.error in .catch().

services/common/src/logger.ts — Exports a Pino logger. Trust HTTP is not using it today (still console.*).

services/common/src/tracing.ts — OpenTelemetry bootstrap (HTTP/Express/pg/redis if wired). Trust server.ts only starts gRPC + HTTP; it does not call initTracing() like some other stacks might.

Net: For this issue, if they want structured logs or consistent request IDs, the natural places are trust http-server / grpc-server (or adopting logger from @common/utils / shared package path you use), and/or relying on gateway + Caddy JSON logs for edge correlation. There is no dedicated “trust interceptor” package.
