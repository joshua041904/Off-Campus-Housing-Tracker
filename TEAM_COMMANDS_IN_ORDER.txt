================================================================================
NOTES ON THE LINEAR RUN (reference — do not paste this whole file into a shell)
================================================================================

• export KUBECONFIG=… — only if kubectl does not see the Colima cluster; omit on k3d/Linux if your kubeconfig is already default.

• trust-dev-root-ca-macos.sh — macOS only; on Linux/WSL import certs/dev-root.pem per LOCAL_TLS_AND_TESTING_GUIDE.md (skip or replace that line).

• export PATH=…curl — Apple Silicon Homebrew; on Intel use:
    export PATH="/usr/local/opt/curl/bin:$PATH"
  Install curl first if needed: brew install curl

• Ollama — in another terminal run ollama serve, then ollama pull llama3.2, then run kubectl set env. Skip Ollama + kubectl set env and add SKIP_K6_ANALYTICS_LISTING_FEEL=1 to preflight for a faster partial run.

• kubectl set env … OLLAMA_BASE_URL — Colima/Docker Desktop Mac; on Linux use your host bridge IP (see GITHUB_PR_DESCRIPTION.txt §4.7).

• sudo sh -c 'echo …' — replace <EXTERNAL_IP> with the LB/ingress IP from kubectl get svc (same guide as hosts).

• k3d / no Colima — skip setup-new-colima-cluster.sh; keep REQUIRE_COLIMA=0 on preflight (already in the line above).

• MetalLB pool — METALLB_POOL=192.168.64.240-192.168.64.250 matches many Colima bridges (Apple Silicon). Wrong subnet → caddy-h3 EXTERNAL-IP stays <pending> and QUIC/HTTP/3 from the host breaks. If eth0 is 192.168.5.x (see colima ssh -- ip -4 addr show eth0 right after setup), use METALLB_POOL=192.168.5.240-192.168.5.250. Omit METALLB_POOL= on setup to auto-detect .240-.250 on the node subnet (scripts/install-metallb-colima.sh). kubectl get svc caddy-h3 and ./scripts/apply-metallb-pool-colima.sh run after bring-up-cluster-and-infra.sh when the LoadBalancer Service exists; apply-metallb-pool-colima.sh fixes pool/L2 and stuck <pending> IPs.

• SSLKEYLOGFILE + QUIC — export SSLKEYLOGFILE before curl --http3 (or before packet capture) so OpenSSL/ngtcp2 writes the NSS key log Wireshark/tshark need (-o tls.keylog_file:…). For rotation suite decrypted frame proof, use ROTATION_H2_KEYLOG=1 on preflight instead of 0 (slower, host k6; see scripts/rotation-suite.sh). Captures: CAPTURE_V2_TLS_KEYLOG or SSLKEYLOGFILE in scripts/lib/packet-capture-v2.sh and scripts/test-packet-capture-standalone.sh.
