Messaging + Kafka architecture (docs/MESSAGING_KAFKA_ARCHITECTURE.md)
--------------------------------------------------------------------
- One-sentence model: Messaging is synchronous (gRPC); Kafka is asynchronous fanout (side-effect propagation).
- Responsibility table: send/store/read via gRPC/Postgres; read receipts via last_read_at; emit event via outbox; analytics/notification/trust consume (messaging service does NOT consume its own topic).
- What Kafka is NOT: chat delivery, UI updates, read receipts, RPC, message broker between users.
- SendMessage flow: gRPC → insert message → insert outbox (MessageSentV1) → commit → publisher → produce to ${ENV_PREFIX}.messaging.events with key = conversation_id.
- Partition key: conversation_id only (6 partitions; ordering per conversation; do not use message_id or random key).
- Payload discipline: no message content in events; MessageSentV1 stays small.
- Exactly-once: effectively exactly-once effect via outbox + idempotent consumers; do NOT add Kafka transactions, EOS, 2PC, sagas, distributed locking.
- Messaging service design: gRPC + Postgres + outbox + publisher + Kafka producer + health; no consumer inside messaging.
- Media split: messaging stores metadata + media_id; Media service handles upload/storage/signed URLs; no media bytes in Kafka/Postgres/gRPC.
- gRPC/HTTP/3: keep unary RPC over HTTP/2; do not attempt gRPC over HTTP/3 manually.
- DLQ: ${ENV_PREFIX}.messaging.dlq; on N consumer failures publish to DLQ, log, alert.
- Observability: metrics for messaging (messages_sent_total, conversation_count, db_latency, kafka_publish_latency), media (uploads_total, upload_failures, upload_bytes_total), notification (notifications_sent_total, failures_total).
- Break tests: referenced as verification of event spine (no loss, at-least-once, idempotent consumers, health).
