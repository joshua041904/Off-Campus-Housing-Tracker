feat(event-layer): codify stability, add intentional break tests, verification-before-freeze

What we are doing
-----------------
We are locking the event layer as production-grade and testable, then freezing it.

1. Stability document (docs/EVENT_LAYER_STABILITY.md)
   - Critical ordering is codified: produce → UPDATE published = true → commit. Never mark published before produce; never commit the update before produce success.
   - Failure scenarios are documented: produce fails → retry; produce succeeds but DB update/commit fails → duplicate, idempotent consumers absorb; update-before-produce → forbidden (event loss).
   - Guarantees are explicit: no lost events, at-least-once delivery, idempotent consumption, deterministic retry.
   - Anti-patterns and freeze criteria are written down. "Do NOT add" list: Kafka transactions, exactly-once producer, saga orchestrators, choreography engines.

2. Verification before freeze (same doc)
   - Design does not freeze the layer; verification does. The doc now states that explicitly and lists the exact steps: implement → run → test → break intentionally → verify idempotency/retry/health → then freeze.
   - The three intentional break tests are specified in full:
     - Test 1 — Kill after produce: inject crash after Kafka produce success, before UPDATE published = true. Expected: row still published = false; restart republishes; consumer idempotency absorbs duplicate.
     - Test 2 — Kill after UPDATE but before commit: simulate transaction rollback. Expected: row still published = false; retry works.
     - Test 3 — Kafka down: publish fails, published remains false, health NOT_SERVING when Kafka required; when Kafka returns, retries succeed.
   - After all three pass, the event layer is considered battle-tested and frozen. Focus then shifts to search/filtering, recommendation engine, UX, observability—not event-layer tweaking.

3. Architecture rules (ARCHITECTURE_RULES.md)
   - Rule 9 — Event layer ordering (outbox) is already in place: publisher must produce → update published → commit; consumers must be idempotent (processed_events, insert event_id before handle, on conflict skip). No Kafka transactions, exactly-once, sagas, or choreography. This makes the event layer a system invariant, not a suggestion.

4. Automated break tests (services/event-layer-verification)
   - New package with vitest that encodes the three break tests as automated tests.
   - Uses in-memory mocks (MockOutboxDb, MockProducer, MockKafkaHealthCheck) so no real Kafka or Postgres is required. The same ordering and failure hooks can be used against real infra for manual or CI integration runs.
   - Test 1: Crash after produce → assert row stays false, produce called once; "restart" (run again without crash) → assert row now true, produce called twice; consumer dedupe → assert only one logical process (duplicate skipped).
   - Test 2: Crash after UPDATE but before commit → assert row stays false (rollback); retry → assert row true and produce called again.
   - Test 3: Kafka down (producer fails) → assert published stays false, health check false; Kafka back → retry succeeds, row true.
   - All six tests pass. Run with: pnpm --filter event-layer-verification test (or from services/event-layer-verification: pnpm test).

Why we are doing this
---------------------
- Most teams never codify ordering, failure cases, and freeze criteria for their event backbone. We are doing it so the system stays predictable and maintainable.
- Freezing by verification (not by design alone) is the correct order of maturity: implement, run, test, break intentionally, verify idempotency and retry and health, then freeze. That is behavior-driven architecture instead of docs-driven wishful thinking.
- Putting Rule 9 in ARCHITECTURE_RULES.md makes the event layer governance-level. It prevents future "let's try Kafka transactions," random choreography experiments, and half-baked exactly-once changes. That is how stable systems are maintained.
- The intentional break tests prove that (1) we never lose events when we follow the order, (2) duplicates are safe because consumers are idempotent, and (3) when Kafka is down we do not mark published and health correctly reflects NOT_SERVING. Once these pass, further complexity in the event layer decreases reliability; the architecture is at equilibrium.

What we are not doing
---------------------
- We are not adding Kafka transactions, exactly-once producer mode, saga orchestrators, or choreography engines. The stability doc and Rule 9 explicitly forbid these unless there is an explicit architecture decision.
- We are not touching the ordering or adding extra layers. The spine is built; the next work is the body: search, recommendations, UX, observability.
