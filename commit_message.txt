perf(lab): add classification + protocol-merit interpretation pipeline from combined CSV

WHY
---
  ‚ÄĘ We already collect collapse-aware, machine-readable rows in `ALL_SERVICES_PROTOCOLS_VU_COMBINED.csv`,
    but we lacked an automated interpretation layer for per-service strategy decisions.
  ‚ÄĘ The previous `make test-current` flow could fail in `make model` because CLI parsing in
    `derive-service-model.js` treated `--pools` values as positional CSV input.
  ‚ÄĘ We need deterministic, repeatable outputs that classify services, identify protocol merit curves,
    summarize collapse behavior, and model only DB-bound services before any new load generation.

WHAT CHANGED
------------
  1) New interpretation builder
     ‚ÄĘ Added `scripts/perf/build-performance-lab.js`.
     ‚ÄĘ Input: combined dataset CSV (for example:
       `bench_logs/ceiling/<stamp>/combined-10/ALL_SERVICES_PROTOCOLS_VU_COMBINED.csv`).
     ‚ÄĘ Output directory: `bench_logs/performance-lab/`.
     ‚ÄĘ Produces:
       - `service-classification.json`
       - `per-service-best-protocol.json`
       - `collapse-summary.json`
       - `protocol-merit.json`
       - `service-models.json` (only DB-bound classes)
       - `final-classification.json`
       - `performance-lab-report.md`

  2) Classification + merit logic
     ‚ÄĘ Computes per-service:
       - best protocol by max pre-collapse RPS
       - collapse spread across protocols
       - protocol RPS variance at matched VUS points
     ‚ÄĘ Assigns service classes via data-driven heuristics:
       - `DB_POOL_LIMITED(_TRANSPORT_SENSITIVE)`
       - `TRANSPORT_SENSITIVE`
       - `BACKEND_BOUND`
       - `CPU_BOUND`
       - `FANOUT_AMPLIFIED`
     ‚ÄĘ Adds protocol merit detection:
       - throughput merit start VUS (A > B by >10%, consistent windows)
       - latency merit start VUS (p95 improvement >15%, consistent windows)
       - stability merit (later collapse VUS)
       - intensity metrics (avg throughput advantage %, avg p95 improvement %)

  3) DB-bound-only modeling + headroom
     ‚ÄĘ Restricts Little‚Äôs Law-style modeling to services classified as DB-bound.
     ‚ÄĘ Computes:
       - `safe_rps = 0.8 * max_rps_pre_collapse`
       - `current_rps_at_vus_20`
       - `headroom_rps`
       - pool utilization and predicted safe RPS by configurable pool sizes.

  4) Makefile integration
     ‚ÄĘ Added target:
       `make performance-lab-interpret CSV=<path-to-combined-csv>`
     ‚ÄĘ Added this target to `.PHONY` and `help`.
     ‚ÄĘ Keeps output deterministic at `bench_logs/performance-lab/`.

  5) CLI parser bug fix
     ‚ÄĘ Updated `scripts/load/derive-service-model.js` argument parsing so flag values
       (e.g. `--pools "10,20,30,40"`) are not mistaken for positional CSV input.
     ‚ÄĘ Prevents failures like:
       `CSV not found: .../10,20,30,40`.
