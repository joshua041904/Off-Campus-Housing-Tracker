perf(lab,make,docs): performance-lab pipeline, reviewer Golden Path, tiered make menu

WHY
---
  ‚ÄĘ Ceiling runs produce `results.csv` and (after flattening) combined CSVs, but teams need a
    single path from raw sweep ‚Üí interpretation artifacts ‚Üí derived pool / ingress guidance without
    hand-rolled shell.
  ‚ÄĘ `derive-service-model.js` previously mis-parsed `--pools` as a positional CSV path.
  ‚ÄĘ Reviewers and operators rely on `GITHUB_PR_DESCRIPTION.txt` for copy/paste flows; the new
    Make targets should be documented next to the existing script-only recipes.
  ‚ÄĘ EXPLAIN coverage for all housing Postgres instances should be one command, consistent with
    `scripts/perf/run-all-explain.sh` (ports 5441‚Äď5448 in the script).
  ‚ÄĘ The PR body read like an infra platform dump; reviewers needed one default validation path,
    a modes table, and explicit ‚Äúadvanced perf‚ÄĚ labels so standard review does not require
    understanding ceiling/capacity tooling.

WHAT CHANGED
------------
  1) Interpretation pipeline (`scripts/perf/build-performance-lab.js`)
     ‚ÄĘ Input: `ALL_SERVICES_PROTOCOLS_VU_COMBINED.csv` (or path via Make).
     ‚ÄĘ Output dir: `bench_logs/performance-lab/`.
     ‚ÄĘ Writes: `service-classification.json`, `per-service-best-protocol.json`,
       `collapse-summary.json`, `protocol-merit.json`, `service-models.json` (DB-bound classes),
       `final-classification.json`, `performance-lab-report.md`.
     ‚ÄĘ Classification, protocol merit (throughput / latency / stability / intensity), headroom
       estimates for DB-bound services.

  2) Combined-10 generation (`scripts/perf/build-combined-10.js`)
     ‚ÄĘ Builds `combined-10/` under a ceiling stamp from that run‚Äôs `results.csv`, including
       `ALL_SERVICES_PROTOCOLS_VU_COMBINED.csv`.

  3) Capacity recommendations (`scripts/capacity/derive-pool-sizes.js`)
     ‚ÄĘ Reads performance-lab JSONs; emits `capacity-recommendations.json`,
       `ingress-tuning.md`, `capacity-dashboard-schema.json`.
     ‚ÄĘ `--min-pool` floor (default 5) avoids unrealistic tiny pool suggestions for low-őľ services;
       wired as `MIN_RECOMMENDED_POOL` in the Makefile `capacity-recommend` target.

  4) Makefile ‚ÄĒ perf / lab / capacity / DB EXPLAIN + menu UX
     ‚ÄĘ Targets: `performance-lab-interpret`, `performance-lab-interpret-latest`,
       `performance-lab-one`, `capacity-recommend`, `capacity-one`, `explain-all-dbs` (see prior

       `scripts/shims`.
     ‚ÄĘ `menu` (default goal): tiered echo groups ‚ÄĒ Core (`up`, `up-fast`, `strict-canonical`,
       `test-current`), Performance / modeling (`collapse-all`, `performance-lab-one`,
       `capacity-one`, `protocol-matrix`), SRE / deep infra (`packet-capture`, `demo-network`,
       `demo-k3d`), plus ‚ÄúAlso common‚ÄĚ and output paths; footer states `make help` is still the
       full flat list (no targets removed).

  5) CLI fix (`scripts/load/derive-service-model.js`)
     ‚ÄĘ Positional CSV and `--pools` flag values no longer collide (avoids ‚ÄúCSV not found: 10,20,‚Ä¶‚ÄĚ).

  6) Documentation (`GITHUB_PR_DESCRIPTION.txt`) ‚ÄĒ runbook + reviewer clarity
     ‚ÄĘ Top-of-doc and section 1: pointer to Reviewer Golden Path.
     ‚ÄĘ Section 2: **Reviewer Golden Path** (`make up-fast` then `make strict-canonical`, first-time

     ‚ÄĘ Review owners: reviewers need not master the full performance lab; validate owned services
       after the Golden Path.

     ‚ÄĘ Sections 4.10, 6.8, 7.5, 7.7: lead with ‚ÄúAdvanced performance tooling ‚ÄĒ not required for
       standard PR review‚ÄĚ (7.7 adds ceiling/collapse/capacity scope note).
     ‚ÄĘ Earlier additions retained: section 7.7 Make target table, `explain-all-dbs`, sweep overrides,
       cross-links to EXPLAIN and perf scripts.
