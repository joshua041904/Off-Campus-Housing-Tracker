feat(och): gateway E2E traffic shaper, Redis weighted budget (Lua), transport-watchdog; Playwright stability

WHY
---
  • Parallel Playwright workers against a dev cluster can stampede the API gateway and backends;
    ad-hoc 503s and timeouts looked like “flaky tests” in GitHub issues but were often real
    concurrency saturation plus brittle UI locators.
  • We need a single place to smooth ingress (semaphore + optional cluster-wide weighted cap in
    Redis) and a signal when the gateway transport path is unhealthy (sidecar → Redis throttle).
  • UI tests that targeted generic inputs across the whole /listings page broke whenever layout
    or duplicate fields changed; success banners without stable hooks were race-prone.
  • TypeScript: the Redis client from createClient() carries module generics that did not match
    a bare RedisClientType import — middleware must accept the same client type the server uses.

WHAT CHANGED
------------
  API gateway (services/api-gateway)
  • E2E traffic shaper (E2E_TRAFFIC_SHAPER*) — bounded concurrency + queue; probes/metrics paths
    skip shaping (gateway-traffic-skip).
  • Cluster-wide weighted budget — redis.eval Lua (atomic try/release); per-route segment weights;
    503 when over cap; fail-open if Redis unavailable.
  • Watchdog throttle poller — reads Redis key set by sidecar; halves shaper capacity when set.
  • server.ts wires shaper + cluster middleware before createHttpConcurrencyGuard when enabled;
    Redis typing uses ReturnType<typeof createClient> for cluster-weight + watchdog modules.

  Transport watchdog (services/transport-watchdog)
  • Small Node service: polls gateway readyz, sets/deletes Redis throttle key with TTL.
  • Dockerfile (build from repo root); pnpm --filter transport-watchdog run build from monorepo root.

  Kubernetes
  • api-gateway deployment — second container transport-watchdog; env for REDIS + probe URL/key.
  • overlays/dev/patches/gateway-traffic-shaper-cluster-weight.yaml — enable shaper, cluster weight,
    watchdog poll on gateway app container.
  • overlays/dev/patches/e2e-high-cap-mode-env.yaml — E2E_HIGH_CAP_MODE=1 on services using
    createHttpConcurrencyGuard (dev / Playwright clusters only).

  Common (services/common)
  • createHttpConcurrencyGuard — E2E_HIGH_CAP_MODE / PLAYWRIGHT_CONCURRENCY_BOOST doubles the
    numeric default when per-service HTTP_MAX_CONCURRENT is unset (explicit env still wins).

  Playwright / webapp (addresses flaky E2E reports)
  • listings/page.tsx — data-testid="listing-created-banner" on the post-create success message.
  • listing-and-analytics-journey.spec.ts — scope the “Post a listing” form via a section that
    contains that heading, so tests survive layout tweaks and deploy lag vs local source; assert
    banner via getByTestId; generous timeouts on create/search/analytics flows.
  • Related specs (listings-filters-maps, system-integrity, analytics.deterministic) — assert
    listing-created-banner and listings error surfaces with stable testids where applicable.
  • webapp/playwright.config.ts — normalizeE2E_API_BASE: HTTPS edge only; reject legacy
    http://127.0.0.1:4020-style bases so pages and API match the same host as CI/docs expect.
  • scripts/webapp-playwright-strict-edge.sh — NODE_EXTRA_CA_CERTS (dev-root.pem) + E2E_API_BASE
    via edge helpers for strict TLS runs against off-campus-housing.test.

  CI
  • .github/workflows/docker-build.yml — matrix entry for transport-watchdog image.

RUN
---
  Rebuild images that embed @common createHttpConcurrencyGuard + gateway + sidecar, load into cluster,
  apply dev overlay (E2E_HIGH_CAP_MODE + traffic patches):

    SERVICES="api-gateway auth-service booking-service listings-service messaging-service \
analytics-service trust-service notification-service" \
      ./scripts/rebuild-och-images-and-rollout.sh
    ./scripts/rebuild-traffic-control-stack.sh   # api-gateway + transport-watchdog only

    kubectl apply -k infra/k8s/overlays/dev

  Local strict Playwright (edge TLS):

    ./scripts/webapp-playwright-strict-edge.sh
