feat(infra): DB schemas, one-shot setup, and proto contract layer (DB + proto + events aligned)

Database layer
--------------
- infra/db: SQL schemas for all six raw-Postgres housing DBs (listings, booking,
  trust, messaging, notification, analytics). Listings: 01-listings-schema-and-tuning.sql
  + 02-listings-pgbench-trigram-knn.sql; each other service: single 01-*-schema.sql.
  Ports 5442–5447 per README; auth (5441) remains Prisma + optional legacy restore.
- scripts/ensure-*-schema.sh: one script per DB (ensure-listings-schema, ensure-booking-schema,
  ensure-messaging-schema, ensure-notification-schema, ensure-trust-schema, ensure-analytics-schema).
  Each applies the corresponding infra/db SQL to the correct port and database.
- scripts/setup-all-dbs.sh: one-shot setup that runs all six ensure scripts in order.
  Usage: PGPASSWORD=postgres ./scripts/setup-all-dbs.sh. Optional DO_DOCKER_UP=1 to
  start all seven Postgres containers before applying schemas. Auth is skipped (Prisma/restore).
- scripts/restore-good-db-settings.sh: applies system-level, database-level, and session-level
  Postgres tuning to all seven instances (localhost 5441–5447). Includes search_path per
  schema (auth, listings, booking, messaging, notification, trust, analytics). Idempotent.
- infra/db/README.md: documents one-shot setup (setup-all-dbs.sh), per-DB apply commands,
  and optional restore-good-db-settings.sh. Port table and schema summaries unchanged.

Proto contract layer (no drift with DB or Kafka)
-----------------------------------------------
- proto/common.proto (infra-grade): Shared types only. No domain messages.
  - Identifiers: UUID (string value).
  - Money: amount_cents (int64) + currency; no floats/decimals.
  - Pagination: PaginationRequest (page, limit), PaginationResponse (total, page, limit).
  - Metadata: RequestMetadata (correlation_id, user_id, role), ResponseMetadata
    (correlation_id, timestamp via google.protobuf.Timestamp). Optional; supports
    tracing and gateway-injected identity.
  - Errors: ErrorDetail (field, issue), ErrorResponse (code, message, details, status),
    StatusCode enum (UNKNOWN, OK, INVALID_ARGUMENT, NOT_FOUND, UNAUTHORIZED, FORBIDDEN,
    CONFLICT, INTERNAL_ERROR). Single structured error format across services.
  - SoftDelete: deleted + deleted_at (Timestamp) for messaging, listings, etc.
  - Empty for RPCs with no body. Import: google/protobuf/timestamp.proto.

- proto/listings.proto: ListingsService — CreateListing, GetListing, SearchListings.
  Messages: CreateListingRequest (user_id, title, description, price_cents, amenities,
  smoke_free, pet_friendly, furnished, effective_from/until), GetListingRequest,
  ListingResponse (listing_id through status, created_at; no search_norm or DB internals),
  SearchListingsRequest/Response. Aligned to listings.listings columns; no implementation leakage.

- proto/booking.proto: BookingService — CreateBooking, ConfirmBooking, CancelBooking, GetBooking.
  Messages: CreateBookingRequest (listing_id, tenant_id, start_date, end_date), confirm/cancel/get
  requests, BookingResponse (booking_id, listing_id, tenant_id, status, created_at). Status
  reflects state machine (CREATED, CONFIRMED, CANCELLED, COMPLETED; DB also has
  pending_confirmation, rejected, expired). Aligned to booking.bookings and Kafka booking.* events.

- proto/trust.proto: TrustService — FlagListing, SubmitReview, GetReputation. Messages:
  FlagListingRequest/FlagResponse, SubmitReviewRequest (booking_id, reviewer_id, rating, comment),
  ReviewResponse, GetReputationRequest, ReputationResponse (user_id, score). No DB internals
  or scoring algorithm exposed. Aligned to trust.listing_flags, trust.reviews, trust.reputation.

- proto/messaging.proto: MessagingService — SendMessage, GetConversation. Messages:
  SendMessageRequest (conversation_id, sender_id, content), MessageResponse (message_id, sent_at),
  GetConversationRequest, ConversationResponse (repeated Message), Message (message_id, sender_id,
  content, sent_at, read). Soft delete and read state handled server-side. Aligned to
  messaging.conversations, messaging.messages, messaging.conversation_participants.

- proto/notification.proto: NotificationService — GetUserPreferences only. Messages:
  GetUserPreferencesRequest (user_id), UserPreferencesResponse (email_enabled, sms_enabled,
  push_enabled). No SendNotification RPC; delivery is Kafka-triggered only. Consumer contract.

- proto/analytics.proto: AnalyticsService — GetDailyMetrics only (read-only). Messages:
  GetDailyMetricsRequest (date), DailyMetricsResponse (new_users, new_listings, new_bookings,
  completed_bookings). No mutating RPCs. Aligned to analytics.daily_metrics.

- proto/auth.proto: AuthService — Register, Login, ValidateToken. Messages: RegisterRequest,
  LoginRequest, AuthResponse (user_id, jwt), ValidateTokenRequest, ValidateTokenResponse
  (user_id, role, valid). Minimal and secure; gateway may validate JWT locally.

- proto/health.proto: Unchanged (grpc.health.v1 Health.Check, Health.Watch).

Design rule: DB = source of truth, proto = public contract, Kafka = externalized state
transition. Emit events after transaction commit; proto models reflect business state, not
DB implementation. common.proto is infra-only; domain-specific types stay in domain protos.

Ports (unchanged): auth 5441, listings 5442, bookings 5443, messaging 5444, notification 5445,
trust 5446, analytics 5447.
