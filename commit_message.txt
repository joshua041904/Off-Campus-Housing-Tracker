feat(perf-lab): protocol happiness matrix, tail-weighted scores, transport dominance

PR #2 (protocol intelligence). PR #1 covered capacity / pool derivation; PR #3 will wire
runtime pool env, ingress stream caps, and policy.

WHY
---
  • After capacity envelopes and derive-pool-sizes.js, we still lacked a deterministic answer
    to: which protocol is best per service, when HTTP/3 beats HTTP/2, and when backend saturation
    masks transport gains.
  • Raw RPS alone mis-ranks protocols when fail_rate or tail amplification is high (e.g. HTTP/2
    at high error throughput).

WHAT CHANGED
------------
  1) scripts/protocol/compute-happiness.js
     • Inputs: --service-model (ceiling service-model.json), --collapse (collapse-summary.json),
       --out-dir; auto-reads capacity-recommendations.json from out-dir when present.
     • Per service: merge duplicate protocol rows by max RPS; compute Throughput, Latency,
       TailRisk (normalized penalty from p95/p50, max/p95, fail_rate), Stability (fail_rate gates),
       Composite = 0.35T + 0.2L + 0.25Tail + 0.2S; winner; envelope @ pool=10; collapse max RPS;
       recommended pool.
     • Transport dominance: λ2/λ3, τ = λ3/λ2 − 1, pool_threshold = λ3/μ_http3, B = μ×pool,
       dominance rule B > λ2×T; h3_transport_unlocked only if τ > 0 AND recommended_pool ≥
       ceil(pool_threshold); τ ≤ 0 forces unlocked false + explanatory note.

  2) Outputs under bench_logs/performance-lab/
     • protocol-happiness-matrix.json
     • protocol-superiority-scores.json
     • protocol-ranking.md

  3) Makefile
     • protocol-happiness target (latest ceiling service-model + perf-lab collapse-summary).
     • capacity-one now runs protocol-happiness after capacity-recommend (pools align with
       dominance math).
     • menu lists protocol-happiness under performance / modeling.

  4) docs
     • GITHUB_PR_DESCRIPTION.txt section 7.7: capacity-one + protocol-happiness.
     • GITHUB_PR_DESCRIPTION_PR2.txt: full PR #2 body for GitHub paste (sample JSON excerpt).

WHAT THIS DOES NOT DO
---------------------
  • No DB pool env changes, no K8s deployment edits, no ingress stream cap changes — analytical
    tooling only (PR #3).

RUN / VALIDATION
----------------
  • node scripts/protocol/compute-happiness.js --service-model <ceiling>/service-model.json \
      --collapse bench_logs/performance-lab/collapse-summary.json --out-dir bench_logs/performance-lab
  • make protocol-happiness / make capacity-one
  • Verified: τ ≤ 0 ⇒ h3_transport_unlocked false + note; τ > 0 and pool ≥ threshold ⇒ true.
