## Colima rebuild cheat sheet (which script?)

Use this **after PR1 / first-time cluster is up** (see **First-Time Run Once** above). Run from **repo root** with **Colima running** and **`kubectl` pointed at your local k3s** so new images actually land in the cluster.

### What “rebuild” means here (local dev)

1. **`docker build`** the service image(s) (often tag `:dev`).
2. **`docker save | colima ssh -- docker load`** so the k3s node can pull the image.
3. **`kubectl rollout restart`** the matching Deployment(s) in namespace **`off-campus-housing-tracker`** (and optionally wait for rollout status).

If you skip these steps after editing code, the browser will still hit **old** pods.

### Two scripts — when to use which

| Script | Builds | Use when |
|--------|--------|----------|
| **`scripts/rebuild-housing-colima.sh`** | **Webapp** (Next.js) **always**, then **selected backend(s)** | You changed anything under **`webapp/`**, or you need a new **Maps API key** baked into the client bundle. Also use if you changed **webapp + one or more backends** in one go. |
| **`scripts/rebuild-och-images-and-rollout.sh`** | **Backend service image(s) only** | You changed **only** Go/Node services under **`services/`** (and **not** the Next.js app). Faster than the housing script because it skips the webapp image. |

### Decision table (what to run)

| You changed | Run | Notes |
|-------------|-----|--------|
| **Webapp** (Next.js), with default **listings-service** | `./scripts/rebuild-housing-colima.sh` **or** `pnpm run rebuild:housing:colima` | Reads **`NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`** from **`webapp/.env.local`** at **build** time (see **`webapp/env.local.template`**). Default backend is **`listings-service`**; override with **`SERVICES=...`**. |
| **One** backend microservice (no webapp) | `SERVICES=<name> ./scripts/rebuild-och-images-and-rollout.sh` **or** `pnpm run rebuild:service:<name>` (see shortcuts below) | Builds **`:dev`** image, loads into Colima, restarts that Deployment. |
| **Several** backend services (no webapp) | `SERVICES="svc-a svc-b" ./scripts/rebuild-och-images-and-rollout.sh` | List every service you modified so all images refresh. |
| **Webapp + one or more backends** | `SERVICES="listings-service auth-service" ./scripts/rebuild-housing-colima.sh` | One entry point: new webapp image **and** listed backends. |

### `pnpm` shortcuts (optional)

From repo root:

- **Webapp + default listings backend:** `pnpm run rebuild:housing:colima`
- **Single backend (examples):** `pnpm run rebuild:service:auth` · `pnpm run rebuild:service:trust` · `pnpm run rebuild:service:listings` · `pnpm run rebuild:service:analytics` · `pnpm run rebuild:service:media` · `pnpm run rebuild:service:search`
- **API gateway only:** `pnpm run rebuild:gateway:rollout`
- **Generic backend script:** `pnpm run rebuild:och:rollout` (same as `rebuild-och-images-and-rollout.sh` with env you pass)

### Copy/paste examples

**A) Only touched the web UI (and maybe listings API) — most common**

```bash
./scripts/rebuild-housing-colima.sh
```

**B) Only touched `auth-service` (no `webapp/` edits)**

```bash
SERVICES=auth-service ./scripts/rebuild-och-images-and-rollout.sh
# or: pnpm run rebuild:service:auth
```

**C) Touched `api-gateway` + `listings-service`, no webapp**

```bash
SERVICES="api-gateway listings-service" ./scripts/rebuild-och-images-and-rollout.sh
```

**D) Touched webapp and need `auth-service` + `listings-service` rebuilt too**

```bash
SERVICES="listings-service auth-service" ./scripts/rebuild-housing-colima.sh
```

### Useful flags (both scripts — see script headers for full list)

```bash
SKIP_LOAD=1 SERVICES=listings-service ./scripts/rebuild-och-images-and-rollout.sh   # build only, no load into Colima
ROLLOUT=0 SERVICES=media-service ./scripts/rebuild-och-images-and-rollout.sh       # load image but skip kubectl restart
WAIT_ROLLOUT=0 SERVICES=auth-service ./scripts/rebuild-och-images-and-rollout.sh     # do not wait on rollout status
SKIP_ROLLOUT=1 ./scripts/rebuild-housing-colima.sh                                 # build+load webapp/backends, no restart
```

**Rebuild after code changes (this issue)**

Quick matrix (full detail: **Colima rebuild cheat sheet** at top of this file):

- **webapp** (+ default `listings-service`) → `./scripts/rebuild-housing-colima.sh` or `pnpm run rebuild:housing:colima`
- **one backend** only → `SERVICES=<name> ./scripts/rebuild-och-images-and-rollout.sh` or `pnpm run rebuild:service:<name>`

**This issue:** After each **code** change you are validating under load, rebuild **only the touched service(s)** — use **one backend** or **several backends** (or **webapp + backends** if you changed `webapp/`). Example: gateway + listings → `SERVICES="api-gateway listings-service" ./scripts/rebuild-och-images-and-rollout.sh`. Load tests (k6) alone do **not** require a rebuild.

**Problem Statement**

This issue targets high tail latency under concurrency (good p50, unstable p95/p99). Focus is system stability under realistic parallel load, not single-endpoint microbenchmarks.

**Disciplined order (PR + perf — read first)**

Do **not** run full-grid tail work before reviewing concurrent PRs that touch the request path (e.g. listings **`fix/listings-validation-response-handling`**). Follow **`docs/perf/PERF_EXECUTION_ORDER_PHASES_A_B_C.md`**: **Phase A** — checkout/diff PR branch, audit **`validation.ts`** + handlers, **`pnpm --filter listings-service build`**, standalone **`k6-listings-health.js`**, compare p95 to baseline; **Phase B** — per-service **`scripts/load/k6-*-concurrency.js`** + **`scripts/perf/k6-dual-service-contention.js`** in isolation; **Phase C** — **`run-housing-k6-edge-smoke.sh`** + **`run-k6-cross-service-isolation.sh`**. Use **`main`** (or your perf branch) as baseline and keep it clear of unreviewed teammate branches until the PR is understood; see **`docs/perf/LISTINGS_VALIDATION_PR_REVIEW_fix-listings-validation.md`**.

**Preconditions**

1. PR1 baseline is merged and first-time setup is complete.
2. Hostname/TLS path works (`off-campus-housing.test`, `certs/dev-root.pem`).
3. Cluster ready (`kubectl get pods -n off-campus-housing-tracker`).

**Load-lab orchestration (baseline mitigations — use before sysctl / “network exhaustion”)**

- **`run-housing-k6-edge-smoke.sh`** defaults: **`K6_ORCHESTRATION_VU_SCENARIO=1`** (messaging + media use **ramping-vus**, not **constant-arrival-rate**, to avoid dropped iterations / VU pile-up through **api-gateway**), **`K6_SUITE_GATEWAY_DRAIN=1`**, **`K6_SUITE_GATEWAY_DRAIN_MAX_MILLICORES=150`**, **`K6_SUITE_POST_DRAIN_SLEEP_SEC=10`**, **`K6_SUITE_KILL_K6_AFTER_BLOCK=1`** ( **`pkill -9 -x k6`** — set **`0`** if another terminal runs k6). Hook order: drain → post-drain sleep → kill → cooldown (see **`scripts/lib/k6-suite-resource-hooks.sh`**).
- If **TIME_WAIT** / **conntrack** stay low and **node CPU** is moderate but **p95** spikes only in the **full suite** → treat as **orchestration interference** through the shared gateway; read **`docs/perf/TAIL_LATENCY_AND_CROSS_SERVICE_ANALYSIS.md`** (“Ruled out in one lab snapshot” + §2.1).

**Execution Plan (deep, step-by-step)**

1. **Capture baseline tail latency (edge grid)**
   ```bash
   SSL_CERT_FILE="$PWD/certs/dev-root.pem" ./scripts/run-housing-k6-edge-smoke.sh
   ```
   Save full output.

2. **Capture resource contention during the same run (second terminal)**
   ```bash
   ./scripts/perf/watch-cluster-contention.sh
   ```
